# False Positive(오탐) 기각 기능

코드 리뷰 에이전트가 잘못 진단한 이슈를 기각(dismiss)하여 Risk 점수를 정확하게 유지하고, 다음 리뷰에서 같은 이슈가 반복 지적되지 않도록 하는 기능.

---

## 배경 및 문제

기존에는 에이전트가 오탐을 냈을 때 처리 방법이 없었다.

- `is_addressed`(처리 완료 체크)는 "코드를 수정해서 해결"을 의미 → 오탐과 의미가 다름
- 그냥 무시하고 merge하면 `risk_score`가 높게 남아 이력 관리가 안 됨
- 다음 push나 `/re-review` 시 에이전트가 동일한 이슈를 또 지적함

---

## 해결 방법 요약

| 문제 | 해결 |
|------|------|
| 오탐인데 risk 점수가 높게 남음 | dismiss 후 남은 활성 이슈 기반으로 `effective_risk` 즉시 재계산 |
| 다음 리뷰에서 같은 이슈 반복 지적 | dismissed 이슈를 `known_false_positives`로 LLM 프롬프트에 주입 |
| 왜 merge됐는지 이력에 안 남음 | `dismissed_reason` 필드에 기각 사유 기록 |

## UI 사용법

### 1. 리뷰 상세 페이지 접근

`/reviews/{reviewId}` 페이지에서 하단 **코멘트** 섹션을 확인한다.

### 2. 오탐 이슈 기각

`issue` 타입 코멘트에만 **"오탐 기각"** 버튼이 표시된다.

```
[ issue ] [ high ] [ src/auth.py ]          오탐 기각  답글
```

버튼 클릭 → 코멘트 바로 아래에 인라인 폼이 열린다.

```
┌──────────────────────────────────────────────────────┐
│ 오탐으로 기각합니다. 사유를 입력하면 다음 리뷰에서    │
│ LLM이 재지적하지 않습니다.                           │
│                                                      │
│ [ 기각 사유 입력 (선택)                           ]  │
│                                        취소  기각 확인 │
└──────────────────────────────────────────────────────┘
```

- 사유는 선택 사항이지만 **입력하는 것을 권장** — 다음 리뷰에서 LLM에 그대로 전달됨
- "기각 확인" 클릭 시 즉시 반영

### 3. 기각 후 화면 변화

**코멘트 카드:**
```
┌──────────────────────────────────────────────────────┐  ← 노란 테두리
│ ☐  [ issue ] [ high ] [ src/auth.py ]   기각 취소  답글
│
│    ~~SQL injection risk in query builder~~           ← 취소선
│
│    오탐으로 기각됨: ORM parameterized query 사용 중
│    · 2026-04-01 14:32
└──────────────────────────────────────────────────────┘
     (카드 전체 반투명)
```

- 노란 테두리 + 반투명 처리
- 본문에 취소선
- 기각 사유와 시각 표시
- 체크박스 비활성화 (기각된 이슈는 "처리 완료" 체크 불가)
- "오탐 기각" → **"기각 취소"** 버튼으로 전환

**리뷰 헤더 (페이지 상단):**
```
[ REQUEST_CHANGES ] [ LOW ] [ push ]  Risk Score: 0.20  ~~0.80~~  [ 오탐 기각 반영됨 ]
```

- Risk 배지와 점수가 재계산된 `effective_risk` 기준으로 갱신됨
- 원본 점수가 취소선으로 나란히 표시됨
- "오탐 기각 반영됨" 배지 노출

### 4. 기각 취소

"기각 취소" 버튼 클릭 → 즉시 원래 상태로 복구, Risk 점수도 재계산

## API 사용법

UI 없이 직접 API를 호출하는 경우.

### 기각

```http
POST /api/reviews/{review_id}/comments/{comment_id}/dismiss
Content-Type: application/json

{
  "reason": "ORM의 parameterized query를 사용 중이라 실제 위험 없음"
}
```

### 기각 취소

```http
DELETE /api/reviews/{review_id}/comments/{comment_id}/dismiss
```

### 재계산된 Risk 확인

```http
GET /api/reviews/{review_id}
```

```json
{
  "risk_score": 0.8,
  "risk_level": "HIGH",
  "effective_risk_score": 0.2,
  "effective_risk_level": "LOW",
  "display_risk_score": 0.2,
  "display_risk_level": "LOW"
}
```

> `display_risk_score` / `display_risk_level` 을 사용하면 항상 올바른 값이 나온다.
> `effective_*`가 있으면 재계산된 값, 없으면 원본 값으로 자동 fallback.

## 동작 원리

### Risk 재계산 로직

dismiss 후 해당 리뷰의 **남은 활성 이슈** (addressed도 dismissed도 아닌 이슈)의 severity를 집계한다.

| 조건 | effective_risk_level | effective_risk_score |
|------|----------------------|----------------------|
| high 이슈 1개 이상 | HIGH | min(0.9, 0.6 + high × 0.1) |
| medium 이슈 2개 이상 | MEDIUM | 0.5 |
| medium 이슈 1개 | MEDIUM | 0.4 |
| 그 외 (low만 있거나 0개) | LOW | 0.2 |

### 다음 리뷰에서의 반영

push 또는 `/re-review` 시 이전 리뷰의 dismissed 이슈가 두 가지 방식으로 처리된다.

1. **미해결 집계에서 제외** — "이전 리뷰 미해결 N개" 카운트에 포함되지 않음
2. **LLM 프롬프트에 재지적 금지로 주입**

```
## 재지적 금지 이슈 (이전에 False Positive로 확인됨)
아래 이슈들은 사람이 검토 후 오탐으로 확인했습니다. 동일한 내용을 다시 지적하지 마세요.
- **src/auth.py**: SQL injection risk (사유: ORM parameterized query 사용 중)
```

## DB 변경 내역 (마이그레이션 0006)

**`review_comments` 테이블 추가 컬럼:**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `severity` | VARCHAR(20) | 이슈 심각도 (high/medium/low). 신규 리뷰부터 저장됨 |
| `is_dismissed` | BOOLEAN | false positive 기각 여부 |
| `dismissed_reason` | TEXT | 기각 사유 |
| `dismissed_by` | VARCHAR(255) | 기각한 사람 (현재 미사용, 향후 인증 연동용) |
| `dismissed_at` | TIMESTAMPTZ | 기각 시각 |

**`reviews` 테이블 추가 컬럼:**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `effective_risk_score` | FLOAT | dismiss 후 재계산된 점수. NULL이면 원본 사용 |
| `effective_risk_level` | VARCHAR(20) | 재계산된 레벨. NULL이면 원본 사용 |

## 현재 제한 사항 및 향후 계획

**현재 제한:**
- `dismissed_by` 필드는 인증 체계가 없어 현재 기록되지 않음
- False positive 패턴은 PR 단위로만 관리됨 (저장소 전체에 누적되지 않음)
- `severity`는 마이그레이션 이후 생성된 리뷰부터만 저장됨 (기존 데이터는 NULL)

**향후 확장 가능 방향:**
- 저장소 단위로 false positive 패턴을 누적하여 반복 오탐 완전 제거
- `dismissed_by` 인증 연동 (dismiss 이력 추적)
