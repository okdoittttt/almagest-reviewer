# 파일 리뷰 흐름 (File Review Flow)

스킬 기반 서브 에이전트 아키텍처를 사용한 코드 리뷰 실행 구조를 설명합니다.

---

## 개요

파일 리뷰는 **이중 병렬 구조**로 동작합니다.

- **1차 병렬**: 변경된 파일들을 동시에 리뷰
- **2차 병렬**: 각 파일에 적용되는 스킬들을 동시에 실행

스킬이 없는 파일은 `system_prompt` 기반 단일 LLM 호출로 fallback됩니다.

```
GitHub Webhook
    ↓
LangGraph 워크플로우
    ↓
[load_skills] → [analyze_intent] → [classify_risk]
    ↓
[review_all_files]  ← 이 문서의 핵심
    ↓
[summarize_review]
    ↓
GitHub PR 코멘트 게시
```

---

## 단계별 상세 설명

### 1단계 — `review_all_files` 노드 진입

LangGraph가 이 노드를 호출하면 다음 순서로 전처리가 이루어집니다.

#### 1-1. 저가치 파일 필터링

lock 파일, 빌드 산출물, 자동 생성 파일 등 리뷰 가치가 낮은 파일을 먼저 제외합니다.

```
package-lock.json  → 스킵 (lock file)
dist/bundle.js     → 스킵 (build artifact)
migrations/        → 통과
app/routers/       → 통과
```

#### 1-2. 위험도 기반 diff 한도 결정

`classify_risk` 노드에서 산출된 위험도(`LOW` / `MEDIUM` / `HIGH`)에 따라 각 파일에 제공할 diff의 최대 문자 수를 결정합니다. 위험도가 높을수록 더 많은 diff를 LLM에 전달합니다.

#### 1-3. 파일 전체를 병렬로 실행

필터링을 통과한 모든 파일을 `asyncio.gather()`로 동시에 처리합니다.

```python
review_tasks = [
    review_single_file(file, idx, ...)
    for idx, file in enumerate(files)
]
results = await asyncio.gather(*review_tasks, return_exceptions=True)
```

---

### 2단계 — `review_single_file` (파일 1개 처리)

각 파일은 독립된 코루틴에서 처리되며, 가장 먼저 **스킬 라우팅**이 수행됩니다.

#### 2-1. 스킬 라우팅 (LLM 호출 없음)

`get_applicable_skills()`는 파일 경로를 각 스킬의 `file_patterns`와 패턴 매칭하여 이 파일에 적용할 스킬 목록을 반환합니다. 추가 LLM 호출 없이 순수한 경로 매칭으로 동작합니다.

| 스킬 | `file_patterns` | 동작 |
|------|----------------|------|
| 보안 점검 | `[]` (비어있음) | 모든 파일에 적용 |
| DB 마이그레이션 검토 | `["**/migrations/**", "**/*.sql"]` | 해당 경로에만 적용 |
| 테스트 품질 검토 | `["**/test_*.py", "**/*_test.py"]` | 테스트 파일에만 적용 |

`file_patterns`가 비어있는 스킬은 항상 모든 파일에 적용됩니다.

#### 2-2. 분기 처리

라우팅 결과에 따라 두 경로 중 하나로 실행됩니다.

---

### 3단계 A — 스킬 서브 에이전트 실행 (스킬 있을 때)

적용 가능한 스킬이 하나 이상 존재하면 각 스킬을 **독립된 서브 에이전트**로 병렬 실행합니다.

```python
skill_tasks = [
    _run_skill_agent(file, skill, diff_max_chars)
    for skill in applicable_skills
]
skill_results = await asyncio.gather(*skill_tasks, return_exceptions=True)
```

#### 서브 에이전트(`_run_skill_agent`)가 받는 것

- 파일 diff (전체 또는 한도 내 일부)
- **해당 스킬의 `name` + `description` + `criteria` 만**

다른 스킬의 기준은 프롬프트에 포함되지 않습니다. 각 에이전트는 오직 자신의 역할에만 집중합니다.

#### 서브 에이전트가 반환하는 것 (최소 아웃풋)

```json
{
  "skill": "보안 점검",
  "verdict": "fail",
  "issues": [
    {
      "severity": "high",
      "message": "SQL 쿼리에 사용자 입력이 직접 삽입되고 있습니다.",
      "suggestion": "파라미터화된 쿼리 또는 ORM을 사용하세요."
    }
  ]
}
```

`verdict` 값:
- `pass` — 이 기준에서 문제 없음
- `warn` — 권장사항 위반 (low/medium 이슈)
- `fail` — 반드시 수정 필요 (high 이슈 존재)

#### 에러 격리

서브 에이전트 하나가 실패해도 다른 에이전트는 계속 실행됩니다. 실패한 에이전트는 `{"verdict": "pass", "issues": []}` 로 안전하게 처리됩니다.

---

### 3단계 B — Fallback 리뷰 (스킬 없을 때)

파일 경로에 매칭되는 스킬이 없으면 `system_prompt` 기반의 단일 LLM 호출로 리뷰를 수행합니다.

```
system_prompt (저장소 설정) + 파일 diff → LLM → 파일 리뷰 결과
```

`system_prompt`도 없으면 최소한의 역할 정의와 JSON 포맷 명세만으로 LLM이 자체 판단합니다. 스킬을 많이 등록할수록 리뷰 정확도와 일관성이 높아지는 구조입니다.

---

### 4단계 — `_aggregate_skill_results` (집계)

모든 스킬 에이전트의 결과를 하나의 파일 리뷰 객체로 합칩니다.

#### 이슈 통합

각 이슈에 어떤 스킬에서 발견됐는지 태그를 추가하고 하나의 목록으로 병합합니다.

```python
for issue in result.get("issues", []):
    issue["skill"] = result.get("skill")  # "보안 점검" 태그 추가
    all_issues.append(issue)
```

#### 파일 상태 결정 (verdict → status)

모든 스킬의 verdict 중 가장 심각한 것을 기준으로 파일 전체의 status를 결정합니다.

| 조건 | 파일 status |
|------|------------|
| `fail` 존재 또는 `high` severity 이슈 존재 | `BLOCKING` |
| `warn` 존재 또는 `medium` severity 이슈 존재 | `MINOR_ISSUES` / `NEEDS_CHANGES` |
| 모두 `pass` | `LGTM` |

---

### 5단계 — `summarize_review` 노드

모든 파일의 리뷰 결과가 모이면 `summarize_review` 노드가 실행됩니다.

- 파일별 status 집계 (LGTM / MINOR_ISSUES / NEEDS_CHANGES / BLOCKING 수)
- 심각도별 이슈 분류 (HIGH / MEDIUM / LOW)
- `system_prompt` 기준에 따라 GitHub PR 코멘트 마크다운 생성
- 최종 판정: `APPROVE` / `REQUEST_CHANGES` / `COMMENT`

## 전체 구조 다이어그램

```
review_all_files
│
├─── 파일A: app/database/migrations/0002.py
│     │
│     ├── get_applicable_skills()
│     │     → [보안 점검, DB 마이그레이션 검토]
│     │
│     ├── asyncio.gather()
│     │     ├── _run_skill_agent(보안 점검)
│     │     │     └── LLM 호출 → {verdict: warn, issues: [...]}
│     │     └── _run_skill_agent(DB 마이그레이션 검토)
│     │           └── LLM 호출 → {verdict: fail, issues: [...]}
│     │
│     └── _aggregate_skill_results()
│           └── {status: BLOCKING, issues: [보안이슈, 마이그레이션이슈]}
│
├─── 파일B: app/models.py  (동시 실행)
│     │
│     ├── get_applicable_skills()
│     │     → []  (패턴 매칭 없음)
│     │
│     └── fallback: create_file_review_prompt + llm.ainvoke()
│           └── {status: LGTM, issues: []}
│
└─── 파일C: app/routers/users.py  (동시 실행)
      │
      ├── get_applicable_skills()
      │     → [보안 점검, API 컨벤션 검토]
      │
      ├── asyncio.gather()
      │     ├── _run_skill_agent(보안 점검)
      │     │     └── LLM 호출 → {verdict: pass, issues: []}
      │     └── _run_skill_agent(API 컨벤션 검토)
      │           └── LLM 호출 → {verdict: warn, issues: [...]}
      │
      └── _aggregate_skill_results()
            └── {status: MINOR_ISSUES, issues: [컨벤션이슈]}

          ↓ 모든 파일 완료 후

summarize_review
└── 3개 파일 결과 종합 → GitHub PR 코멘트 생성 → REQUEST_CHANGES
```

## 예시

### 시나리오

Django 프로젝트에 PR이 올라옴. 변경 파일은 3개:

- `users/migrations/0003_add_phone.py`
- `users/views.py`
- `tests/test_users.py`

저장소에 등록된 스킬:

| 스킬 이름 | 설명 | `file_patterns` |
|----------|------|----------------|
| 보안 점검 | SQL Injection, 인증 우회, 민감정보 노출 점검 | `[]` (전체 적용) |
| 마이그레이션 안전성 | 데이터 손실 가능성, 롤백 가능성 점검 | `["**/migrations/**"]` |
| 테스트 커버리지 | 핵심 시나리오 누락 여부, assert 품질 점검 | `["**/test_*.py", "**/*_test.py"]` |

---

### 실행 흐름

**`users/migrations/0003_add_phone.py`**
- 적용 스킬: `보안 점검`, `마이그레이션 안전성`
- 병렬 실행:
  - 보안 점검 → `{verdict: pass}`
  - 마이그레이션 안전성 → `{verdict: warn, issues: [{severity: medium, message: "phone 컬럼에 NOT NULL 제약이 있는데 기본값이 없습니다. 기존 데이터 마이그레이션 시 오류 발생 가능합니다.", suggestion: "default='' 또는 null=True로 설정하거나 데이터 채움 migration을 선행하세요."}]}`
- 집계 결과: `status: MINOR_ISSUES`

**`users/views.py`**
- 적용 스킬: `보안 점검` (file_patterns 없음 → 전체 적용)
- 단일 실행:
  - 보안 점검 → `{verdict: fail, issues: [{severity: high, message: "request.GET['phone']을 쿼리에 직접 사용하고 있습니다.", suggestion: "ORM filter() 또는 파라미터화된 쿼리를 사용하세요."}]}`
- 집계 결과: `status: BLOCKING`

**`tests/test_users.py`**
- 적용 스킬: `보안 점검`, `테스트 커버리지`
- 병렬 실행:
  - 보안 점검 → `{verdict: pass}`
  - 테스트 커버리지 → `{verdict: warn, issues: [{severity: low, message: "phone 번호 형식 유효성 검사 테스트가 없습니다.", suggestion: "유효하지 않은 phone 형식에 대한 케이스를 추가하세요."}]}`
- 집계 결과: `status: MINOR_ISSUES`

---

### 최종 결과

`summarize_review`가 위 3개 결과를 종합:

```
판정: REQUEST_CHANGES

주요 이슈:
- [HIGH] users/views.py — SQL Injection 위험 (보안 점검)
- [MEDIUM] users/migrations/0003_add_phone.py — NOT NULL 제약 위험 (마이그레이션 안전성)
- [LOW] tests/test_users.py — 테스트 케이스 누락 (테스트 커버리지)
```

## 핵심 설계 원칙

| 원칙 | 내용 |
|------|------|
| **천장 없는 확장** | 스킬을 추가할수록 리뷰 깊이가 선형적으로 증가 |
| **과금 최적화** | 스킬 라우팅은 LLM 없이 패턴 매칭으로 처리 |
| **격리와 안정성** | 스킬 하나의 실패가 전체 리뷰를 중단시키지 않음 |
| **자유도 최대화** | system_prompt + 스킬로 리뷰 방향을 완전히 사용자가 정의 |
| **fallback 보장** | 스킬이 없어도 system_prompt 기반 리뷰가 항상 실행됨 |
